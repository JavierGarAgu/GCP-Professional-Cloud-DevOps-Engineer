

# Google Cloud Professional Cloud DevOps Engineer Lab

# Question - Secure Production Deployments Using Binary Authorization

---

## Introduction

This repository contains a hands-on laboratory created while preparing for the **Google Cloud Professional Cloud DevOps Engineer** certification.

The goal of this lab is to understand how to secure production deployments in **Google Kubernetes Engine (GKE)** by ensuring that only container images created by a trusted CI/CD pipeline can run in production environments.

The exam question focuses on a common security requirement in modern DevOps environments:

> You are managing the production deployment to a set of Google Kubernetes Engine (GKE) clusters. You want to make sure only images which are successfully built by your trusted CI/CD pipeline are deployed to production. What should you do?

Options:

A. Enable Cloud Security Scanner on the clusters.

B. Enable Vulnerability Analysis on the Container Registry.

C. Set up the Kubernetes Engine clusters as private clusters.

D. Set up the Kubernetes Engine clusters with Binary Authorization.

---

# Correct Answer

## D - Set up the Kubernetes Engine clusters with Binary Authorization.

---

# Why?

The correct answer is **D** because **Binary Authorization** is the Google Cloud service designed to control which container images are allowed to run inside GKE clusters.

Binary Authorization implements a **deployment-time security control**.

Instead of trusting every image stored inside a registry, GKE can verify that an image has passed specific security requirements before allowing Kubernetes to deploy it.

In this scenario, the requirement is:

> Only images successfully built by the trusted CI/CD pipeline should be deployed to production.

The solution is to:

1. Build the container image using Cloud Build.
2. Store the image in Artifact Registry.
3. Generate a cryptographic signature for the image.
4. Create an attestation proving that the image was produced by the trusted pipeline.
5. Configure Binary Authorization in GKE.
6. Block every image that does not contain a valid attestation.

The deployment flow becomes:

```
Developer
    |
    |
    v
GitHub Repository
    |
    |
    v
Cloud Build Pipeline
    |
    |
    +----------------+
    |                |
    v                |
Build Image          |
    |                |
    v                |
Artifact Registry    |
    |                |
    v                |
Calculate Digest     |
    |                |
    v                |
Sign Image           |
    |
    v
Create Attestation
    |
    v
GKE Deployment Request
    |
    v
Binary Authorization Verification
    |
    +----------------------+
    |                      |
    v                      v
Valid Signature       Invalid Image

Deploy               Block
```

# Why the other answers are incorrect?

---

## A - Enable Cloud Security Scanner on the clusters

This option is incorrect.

Cloud Security Scanner is focused on detecting vulnerabilities in web applications.

It can identify issues such as:

* Cross-site scripting vulnerabilities.
* Incorrect web application configurations.
* Security weaknesses in HTTP applications.

However, it does not verify whether a container image was created by a trusted CI/CD pipeline.

The problem is not:

> "Is this application vulnerable?"

The problem is:

> "Can we prove that this image was generated by our trusted build process?"

Binary Authorization solves this problem.

---

## B - Enable Vulnerability Analysis on the Container Registry

This option is incorrect.

Vulnerability Analysis scans container images looking for known vulnerabilities inside packages and dependencies.

For example:

```
Container Image
        |
        |
        v
Scan Packages

openssl 1.1.1  -> Vulnerability detected
python 3.10    -> No vulnerability
nginx          -> Vulnerability detected
```

This provides information about image security, but it does not answer the main requirement.

A perfectly clean image could still come from an untrusted source.

Example:

```
Image A

Built by:
Trusted Cloud Build Pipeline

Vulnerabilities:
0


Image B

Built by:
Unknown Developer Laptop

Vulnerabilities:
0
```

Both images have zero vulnerabilities, but only Image A should be deployed.

Binary Authorization validates the origin of the image.

---

## C - Set up Kubernetes Engine clusters as private clusters

This option is incorrect.

Private GKE clusters improve network security by removing public access from cluster nodes.

They help protect:

* Kubernetes nodes.
* Internal workloads.
* Cluster communication.

However, they do not verify container image integrity.

A private cluster could still deploy a malicious image if the deployment process allows it.

Example:

```
Private GKE Cluster

        |
        |
        v

kubectl apply deployment.yaml

        |
        |
        v

Malicious Image
        |
        |
        v

Running Container
```

Binary Authorization prevents this scenario.

# Binary Authorization Concepts

Before understanding the implementation, it is important to understand the main components involved in Binary Authorization.

Binary Authorization is based on a simple security principle:

> Do not trust a container image only because it exists in a registry. Trust it only if it can prove where it was created and who approved it.

The system uses several concepts:

* Container image digest
* Attestation
* Attestor
* Note
* Cryptographic key
* Policy

---

# Container Image Digest

A Docker image can have different identifiers.

The most common one is the image tag:

```text
webapp:latest
```

or:

```text
webapp:v1.0
```

However, tags can change.

Example:

```text
webapp:latest

Today:
sha256:111111111111

Tomorrow:
sha256:222222222222
```

The tag points to a different image over time.

For security purposes, Binary Authorization uses the image digest.

A digest is an immutable identifier:

```text
webapp@sha256:2b3785591bfaa929806d58f2bd1b4d49d13a6acddfca03e9d3a2d3b4287e23ea
```

This represents exactly one container image.

The deployment flow is:

```text
Build Image

     |
     |
     v

Calculate Digest

     |
     |
     v

Sign Digest

     |
     |
     v

Deploy Same Digest
```

The signature is not attached to a tag.

It is attached to the exact image content.

---

# Attestation

An attestation is a security statement that says:

> This image was verified and approved by a trusted process.

In this laboratory, the attestation means:

```text
Cloud Build successfully created this image.
```

The attestation contains information about:

* The image digest.
* The attestor that approved it.
* The cryptographic signature.

Example conceptually:

```json
{
  "image": "webapp@sha256:xxxx",

  "attestor": "trusted-cloudbuild",

  "statement": "Built by trusted CI/CD pipeline"
}
```

Without this attestation, GKE will reject the image.

---

# Attestor

An attestor represents the trusted entity that approves images.

In this laboratory:

```text
trusted-cloudbuild
```

means:

> Images signed by the trusted Cloud Build pipeline.

The attestor does not create signatures.

It verifies them.

The relationship is:

```text
Cloud Build
     |
     |
     v

Creates Signature

     |
     |
     v

Attestation

     |
     |
     v

Attestor

     |
     |
     v

Binary Authorization Policy
```

---

# Note

A Binary Authorization attestor uses a Container Analysis Note.

The note describes what the attestor represents.

In this laboratory:

```text
trusted-cloudbuild-note
```

contains:

```text
Trusted Cloud Build Images
```

The note is basically metadata explaining:

> This attestor represents images produced by our trusted build system.

---

# Cloud KMS Signing Process

The signing process uses asymmetric cryptography.

The laboratory creates:

```text
Cloud KMS Key Ring

binaryauth-keyring

        |
        |
        v

Crypto Key

binaryauth-key

        |
        |
        v

Crypto Key Version

1
```

The key algorithm:

```text
EC_SIGN_P256_SHA256
```

uses:

* Elliptic Curve cryptography.
* SHA256 hashing.

The process:

```text
Container Image

       |
       |
       v

Image Digest

       |
       |
       v

Cloud Build requests signature

       |
       |
       v

Cloud KMS signs digest

       |
       |
       v

Signature returned

       |
       |
       v

Binary Authorization Attestation
```

The important security property:

The private key never leaves Cloud KMS.

---

# Binary Authorization Policy

The policy defines what happens when a deployment occurs.

The configured policy is:

```yaml
defaultAdmissionRule:
  evaluationMode: REQUIRE_ATTESTATION
  enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG

  requireAttestationsBy:
    - projects/devops-cert-labs/attestors/trusted-cloudbuild
```

Meaning:

```text
Every container image
        |
        |
        v

Does it have a valid attestation?

        |
        |
   +----+----+

   YES       NO

   |          |

 Allow      Block
 Deploy     Deployment
```

The important part:

```yaml
evaluationMode: REQUIRE_ATTESTATION
```

forces GKE to verify the image.

And:

```yaml
enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG
```

means:

* Block unauthorized deployments.
* Generate audit logs.



```bash
--artifact-url=image@digest

--attestor=trusted-cloudbuild

--keyversion-key=binaryauth-key
```

```text
Image Digest

        +

Cloud KMS Signature

        +

Attestor Information


             |

             v

```

Now the image is trusted.

The deployment uses the digest.

Not:


But:


This guarantees that Kubernetes deploys the exact signed image.

```hcl
resource "google_container_cluster" "cluster"
```

Cluster:

```text
binaryauth-cluster
```

Zone:

```text
europe-west1-b
```

---

# Binary Authorization Enabled In GKE

The most important configuration:

```hcl
binary_authorization {

  evaluation_mode = "PROJECT_SINGLETON_POLICY_ENFORCE"

}
```

This activates Binary Authorization.

The meaning:

```text
Every deployment request

        |

        v

Check project Binary Authorization policy

        |

        v

Allow or reject
```

The cluster uses a dedicated node pool:

```hcl
resource "google_container_node_pool" "primary_pool"
```

Configuration:

```text
Machine:

e2-medium


Nodes:

2


Disk:

30GB pd-standard
```

The nodes use:

```text
gke-node-sa
```

as their identity.

---

# Workload Identity

The cluster enables:

```hcl
workload_identity_config {

 workload_pool = "devops-cert-labs.svc.id.goog"

}
```

Workload Identity allows Kubernetes workloads to use Google identities securely.

Instead of storing:

```text
Service account keys
```

inside containers, workloads authenticate using Google-managed identities.

# Kubernetes Provider

Terraform also manages Kubernetes resources.

The provider:

```hcl
provider "kubernetes"
```

connects Terraform with the new GKE cluster.

It obtains:

* Cluster endpoint.
* Authentication token.
* CA certificate.

This allows Terraform to create:

```text
Namespace

Deployments

Services

ConfigMaps
```

# Production Namespace

Terraform creates:

```hcl
resource "kubernetes_namespace" "production"
```

Namespace:

```text
production
```

The application is isolated there.

The final Kubernetes structure:

```text
GKE Cluster

      |

      v

production namespace

      |

      +---- Deployment

      |

      +---- Pods

      |

      +---- Service
```
