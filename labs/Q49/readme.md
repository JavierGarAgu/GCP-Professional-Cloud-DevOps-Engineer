# GCP Professional Cloud DevOps Engineer - Preemptible VM Lab

## Exam Question

You need to reduce the cost of virtual machines (VM) for your organization. After reviewing different options, you decide to leverage preemptible VM instances.

Which application is suitable for preemptible VMs?

**A.** A scalable in-memory caching system.

**B.** The organization's public-facing website.

**C.** A distributed, eventually consistent NoSQL database cluster with sufficient quorum.

**D.** A GPU-accelerated video rendering platform that retrieves and stores videos in a storage bucket.

**Correct Answer: D**

---

# Explanation

Preemptible VMs (now called Spot VMs in Google Cloud) are designed for workloads that can tolerate interruptions. Google can terminate these virtual machines at any time, so they should only be used for applications that can safely restart their work.

A video rendering platform is an excellent example because:

- The original video is stored in Cloud Storage.
- The rendered video is uploaded back to Cloud Storage.
- If the VM is terminated during rendering, another VM can start the same job again.
- No important data is lost because everything is stored outside the VM.

The other options are not good candidates:

- An in-memory cache loses data when the VM stops.
- A public website should remain available for users.
- A database cluster should not continuously lose nodes because it can reduce performance and availability.

# Lab Overview

This lab demonstrates a batch processing architecture using a preemptible (Spot) Compute Engine virtual machine.

The VM automatically downloads a video from Cloud Storage, renders it using FFmpeg, and uploads the processed video back to another Cloud Storage bucket.

This architecture is fault tolerant because all important files are stored in Cloud Storage instead of the virtual machine.

---

# Architecture

```text
                 sample.mp4
                      │
                      ▼
        Cloud Storage (Input Bucket)
                      │
                      ▼
      Preemptible / Spot Compute Engine VM
              Startup Script
                    FFmpeg
             Video Processing
                      │
                      ▼
       Cloud Storage (Output Bucket)
                  output.mp4
```

# Terraform Resources

The `main.tf` creates the complete environment automatically.

## Google Cloud APIs

The configuration enables the required Google Cloud services:

- Compute Engine API
- Cloud Storage API

These services are required before creating virtual machines and storage buckets.

---

## Service Account

A dedicated service account is created for the virtual machine.

The service account receives the following IAM role:

- `roles/storage.objectAdmin`

This permission allows the VM to:

- Download videos from the input bucket.
- Upload rendered videos to the output bucket.

---

## Cloud Storage

Terraform creates two storage buckets.

### Input Bucket

Stores the original video file.

Example:

```
sample.mp4
```

### Output Bucket

Stores the rendered video generated by FFmpeg.

Example:

```
output.mp4
```

Using Cloud Storage ensures that data is not lost if the VM is terminated.

---

## Compute Engine VM

Terraform creates one Compute Engine instance configured as a preemptible (Spot) VM.

The VM includes:

- Ubuntu or Debian operating system
- Public IP address
- Service account
- Startup script
- Automatic software installation

Because the VM is preemptible, Google may terminate it at any time, making it significantly cheaper than a standard virtual machine.

---

## Startup Script

The startup script performs the complete workflow automatically.

It:

1. Updates the operating system.
2. Installs FFmpeg.
3. Downloads the original video from Cloud Storage.
4. Converts the video to 720p.
5. Uploads the rendered video to the output bucket.
6. Writes execution logs.

No manual intervention is required after the VM starts.

# Why This Architecture Works

The virtual machine contains no important data.

The original video remains in Cloud Storage.

The rendered video is uploaded to Cloud Storage after processing.

If Google terminates the preemptible VM during rendering, a new VM can process the same video again without losing any files.

This makes video rendering an ideal workload for preemptible virtual machines.
