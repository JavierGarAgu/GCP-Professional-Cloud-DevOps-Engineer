# Q35 - Cloud Build authentication with GKE using kubectl

## Overview

This exercise explains how Cloud Build can deploy new container images to Google Kubernetes Engine (GKE) using the `kubectl` builder.

The exam question asks:

> You use Cloud Build for CI/CD pipelines. You want to use the kubectl builder to deploy new images to GKE. You need to authenticate to GKE while minimizing development effort. What should you do?

The correct answer is:

**A - Assign the Container Developer role to the Cloud Build service account.**

The reason is that Cloud Build already executes builds using a service account identity. The recommended approach is to grant the required IAM permissions to this service account instead of creating credentials manually or adding authentication steps.

# Cloud Build Service Account

The most important part related to Q35 is the Cloud Build service account:

```hcl
resource "google_service_account" "cloudbuild_sa"
```

Cloud Build runs using this identity.

The service account receives permissions:

```hcl
roles/artifactregistry.writer
roles/logging.logWriter
roles/container.admin
```

The important permission for the Q35 exam question is:

```
roles/container.developer
```

because it allows Cloud Build to interact with GKE and deploy workloads using kubectl.

The laboratory uses:

```
roles/container.admin
```

which works but is more permissive than necessary.

A production implementation should replace:

```hcl
roles/container.admin
```

with:

```hcl
roles/container.developer
```

following the principle of least privilege.

# cloudbuild.yaml explanation

The Cloud Build pipeline executes these steps:

## 1. Build Docker image

The first step creates the container image:

```yaml
- name: gcr.io/cloud-builders/docker
```

The image receives a version based on the commit SHA:

```
image:$SHORT_SHA
```

This creates immutable versions.

---

## 2. Push image to Artifact Registry

The next step uploads the image:

```
docker push
```

Now GKE can download the new application version.

---

## 3. Retrieve GKE credentials

The pipeline executes:

```bash
gcloud container clusters get-credentials
```

This configures kubectl to communicate with the cluster.

The important point is that Cloud Build does not need:

* Service account JSON keys
* Manual authentication scripts
* Additional credential steps

Google Cloud automatically uses the Cloud Build service account.

---

## 4. Deploy using kubectl

The final step uses:

```yaml
- name: gcr.io/cloud-builders/kubectl
```

and executes:

```bash
kubectl apply -f k8s/
```

The Kubernetes manifests are applied to GKE and the new image is deployed.

# Why the correct answer is A

The question asks for the solution with the minimum development effort.

The correct flow is:

```
Cloud Build
      |
      |
Cloud Build Service Account
      |
      |
roles/container.developer
      |
      |
GKE API
      |
      |
kubectl deployment
```

This uses native Google Cloud IAM authentication.

---

# Why the other answers are incorrect

## B - Specify the role in cloudbuild.yaml

Incorrect.

Cloud Build YAML only defines execution steps.

IAM permissions are managed separately using Google Cloud IAM.

---

## C - Create another service account

Incorrect.

Cloud Build already has an identity.

Creating another service account adds unnecessary configuration and maintenance.

---

## D - Retrieve service account credentials manually

Incorrect.

Manual credentials are unnecessary and less secure.

Google Cloud services should use IAM-based authentication.

---

# Conclusion

The Q14 laboratory demonstrates the same core concept tested in Q35.

Cloud Build can deploy to GKE securely because it uses its own service account identity.

The recommended solution is:

```
Assign roles/container.developer
to the Cloud Build service account
```

This provides the required access for kubectl deployments while following the principle of least privilege.
